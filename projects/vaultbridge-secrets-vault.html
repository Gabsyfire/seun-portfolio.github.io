Executive Summary

Managed Service Providers handle credentials for dozens — sometimes hundreds — of client environments every day. Shared spreadsheets, sticky notes, and ad-hoc password managers create serious security and compliance exposure. VaultBridge was built to solve that problem at its root: a purpose-built, multi-tenant secrets vault that gives each MSP client a fully isolated workspace for storing, sharing, and auditing passwords and sensitive credentials.

I designed and developed VaultBridge end to end — from the encrypted data layer and RESTful API through to the React front-end and Azure infrastructure. The platform uses AES-256-GCM envelope encryption backed by Azure Key Vault, role-based access control with four distinct permission tiers, and a complete audit trail that logs every credential access, modification, and share event. Authentication supports both local accounts with TOTP two-factor and Microsoft Entra ID single sign-on, giving MSPs the flexibility to integrate with their existing identity infrastructure.

The result is a production-grade platform that replaces insecure, fragmented credential management with a centralised, encrypted, and auditable solution — purpose-built for the multi-tenant reality of managed services.

Background & Context

The managed services industry operates on trust. Clients hand over administrative access to their most critical systems — Active Directory, cloud consoles, network appliances, SaaS platforms — and expect those credentials to be handled with the utmost care. In practice, the reality is often far less reassuring. Many MSPs still store client passwords in shared Excel files, team-accessible password managers with no tenant isolation, or worse, in email threads and messaging apps.

The risks are well documented. A single compromised spreadsheet can expose every client's credentials simultaneously. Without audit logging, there is no way to determine who accessed which password and when — a critical gap during incident response or compliance reviews. Role-based access is typically non-existent, meaning junior technicians have the same visibility into sensitive credentials as senior engineers and account managers.

I identified this gap through direct experience working with MSP environments and set out to build a solution that treats credential management as a first-class security concern rather than an afterthought. VaultBridge was designed from the ground up with multi-tenancy, encryption at rest, granular access control, and comprehensive auditing as foundational requirements — not bolt-on features.

Methodology & Approach

VaultBridge was developed using a structured, security-first methodology. Each phase was designed to validate assumptions early and build confidence in the platform's security posture before adding complexity.

1. Threat Modelling

Before writing a line of code, I mapped the threat landscape for a multi-tenant credential vault. I identified the primary attack vectors — compromised admin accounts, SQL injection, insecure direct object references, privilege escalation between tenants, and encryption key mismanagement — and defined mitigations for each. This threat model directly informed the architecture decisions that followed.

2. Architecture Design

I adopted Clean Architecture principles to enforce separation of concerns across four layers: Domain, Application, Infrastructure, and API. The multi-tenant isolation model uses Entity Framework Core global query filters combined with SQL Server's SESSION_CONTEXT to guarantee that database queries are scoped to the correct tenant at both the application and database levels. This dual-layer approach eliminates an entire class of data leakage vulnerabilities.

3. Encryption Implementation

The encryption layer follows the envelope encryption pattern. Each credential field is encrypted with a unique data encryption key (DEK) using AES-256-GCM. The DEK itself is wrapped using a key encryption key (KEK) stored in Azure Key Vault with Hardware Security Module backing. This design means that even if the database is fully compromised, the credential data remains encrypted and the keys to decrypt it never leave Key Vault.

4. Identity & Access Control

Authentication was implemented with dual support: local accounts using BCrypt password hashing with TOTP-based two-factor authentication, and Microsoft Entra ID integration via MSAL for organisations that prefer federated identity. JWT access tokens are issued with a 15-minute expiry and include tenant-scoped role claims, ensuring that each API request is authorised against the correct tenant context.

5. Infrastructure & Deployment

The entire Azure infrastructure is defined in Terraform modules — Azure Container Apps, Azure SQL with transparent data encryption, Azure Key Vault, Log Analytics, and Front Door. A GitHub Actions CI/CD pipeline handles automated builds, test execution, Terraform validation, and deployment to the production environment on every merge to main.

6. Security Hardening

A dedicated hardening pass addressed twelve compliance areas: deny-by-default authentication, ownership enforcement, secret management, rate limiting, CORS configuration, input validation, password policy, token scoping, CSRF protections, security headers, audit logging coverage, and infrastructure safe defaults. Every finding was remediated and verified with automated tests.

Solution Architecture

VaultBridge follows a layered architecture with strict separation between the API surface, business logic, and infrastructure concerns. The ASP.NET Core 8 API serves as the entry point, handling authentication, authorisation, and request routing. Business rules — credential encryption, tenant resolution, role enforcement, and audit logging — are encapsulated in the Application and Domain layers, ensuring they remain independent of any specific framework or database technology.

The React 18 front-end communicates exclusively through the RESTful API, with JWT tokens stored in session storage and transmitted via Authorization headers. The single-page application is embedded within the API's static file serving, eliminating CORS complexity in production and simplifying the deployment model to a single container.

At the infrastructure level, Azure Container Apps provides the hosting platform with automatic scaling, while Azure SQL handles persistent storage with transparent data encryption and row-level tenant isolation via SESSION_CONTEXT. Azure Key Vault manages the key encryption keys, and Log Analytics aggregates diagnostic and audit data for operational visibility.

Lessons Learned

What Worked Well

The decision to implement tenant isolation at two layers — EF Core query filters at the application level and SESSION_CONTEXT at the database level — proved invaluable. It provides defence in depth: even if a bug bypasses the application-layer filter, the database-layer constraint prevents cross-tenant data access. This pattern gave genuine confidence during security testing and would be my default approach for any future multi-tenant system.

Using envelope encryption with Azure Key Vault was another strong decision. By never exposing the key encryption key outside of Key Vault, the blast radius of a database compromise is dramatically reduced. The key management complexity is handled by the cloud service, and rotation can be performed without re-encrypting every credential — only the wrapped DEKs need to be re-wrapped.

What I Would Improve

The initial implementation relied too heavily on manual security review. In hindsight, I would integrate automated security scanning — static analysis, dependency vulnerability checks, and policy-as-code for Terraform — into the CI pipeline from day one rather than running a dedicated hardening pass after the core build was complete. I would also implement a credential rotation reminder system that proactively alerts MSPs when stored passwords approach a configurable age threshold, reinforcing good hygiene practices at the platform level.

Client Value

VaultBridge addresses a genuine, high-stakes problem that affects virtually every Managed Service Provider. The platform eliminates the single most dangerous practice in MSP credential management — storing client passwords in unencrypted, shared documents — and replaces it with a solution that meets enterprise security expectations without enterprise-level complexity.

For MSP leadership, the value is risk reduction and compliance readiness. The immutable audit trail satisfies the evidence requirements of frameworks like Cyber Essentials, ISO 27001, and SOC 2. The role-based access model enforces the principle of least privilege, ensuring that technicians only access the credentials they need for their current task. And the multi-tenant architecture means that a security incident in one client environment cannot cascade to another.

For the engineers using the platform daily, the value is operational efficiency. Credentials are accessible through a clean, responsive interface with one-click reveal and copy. Secure sharing eliminates the need for out-of-band password exchanges. And the integration with Microsoft Entra ID means one less set of credentials to manage — teams can log in with the same identity they use for everything else.

The platform is built to scale. New tenants can be provisioned in seconds, the infrastructure auto-scales on Azure Container Apps, and the Terraform-defined infrastructure ensures that environments are reproducible and consistent. VaultBridge is not a proof of concept — it is a production-ready platform designed to grow with the MSPs that adopt it.
