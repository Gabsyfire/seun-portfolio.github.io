Executive Summary

Credential management is one of the most critical — and most neglected — challenges facing Managed Service Providers. MSP engineers routinely handle hundreds of client passwords, API keys, and secrets, yet many teams still rely on shared spreadsheets, sticky notes, or consumer-grade password managers that lack multi-tenant isolation.

VaultBridge is an enterprise password and secrets vault I designed and built specifically for the MSP operating model. It provides full multi-tenant isolation, encrypted storage, role-based access control, and a complete audit trail — all behind a clean, modern interface with Microsoft SSO integration. The system ensures that every credential access is logged, every secret is encrypted at rest, and every client's data is strictly separated from every other.

Background & Context

As an MSP, our engineers need instant access to client credentials across dozens of environments — from Active Directory admin accounts to cloud API keys to VPN configurations. Before VaultBridge, these credentials lived across a fragmented landscape: some in shared Excel files, some in email threads, some memorised, and some stored in tools that offered no multi-tenant separation or audit capability.

This created real risk: no visibility into who accessed what, no assurance that a departing engineer's access was revoked, and no compliance-ready audit trail. We needed a solution built for how MSPs actually work — one that treats tenant isolation as a first-class concern and makes secure credential sharing frictionless rather than painful.

Objectives

The primary goal was to build a secure, multi-tenant vault that MSP teams could adopt without friction. Specifically: eliminate unencrypted credential storage across the organisation, provide per-client tenant isolation so no engineer can accidentally access the wrong client's secrets, implement full audit logging for compliance and incident response, integrate with Microsoft identity for seamless SSO, and deliver an intuitive UI that engineers would actually want to use daily.

Architecture

VaultBridge follows a layered security architecture. The frontend communicates via a RESTful API to the vault service layer, which handles encryption, access control evaluation, and tenant context resolution before interacting with the encrypted datastore. Authentication flows through Microsoft Identity Platform for SSO, with a fallback email/password option for external users.

Client App → Auth + API Gateway → Vault Service → Encrypted Store

Every request is authenticated, authorised against the tenant-scoped RBAC policy, and logged before any secret is decrypted and returned. Secrets are encrypted at rest using AES-256, and tenant boundaries are enforced at the data layer — not just the application layer.

Key Features

🔐

Multi-Tenant Vault

Complete data isolation per client. Each tenant's secrets are encrypted independently, ensuring no cross-contamination between client environments.

🪪

Microsoft SSO

One-click sign-in via Microsoft Identity Platform. Engineers use their existing corporate credentials — no additional passwords to manage.

🛡

Role-Based Access Control

Granular permissions per tenant. Admins control who can view, edit, or share secrets within each client's vault.

📋

Full Audit Trail

Every credential access, modification, and share event is logged with user, timestamp, and context — ready for compliance reviews.

🔑

Encrypted Storage

AES-256 encryption at rest for all secrets. Decryption occurs only at the point of authorised access, never in transit or cache.

⚡

Quick Access UI

Clean, modern interface designed for daily use. Copy-to-clipboard, search, and categorisation make credential retrieval instant.

Results & Impact

100% Credentials encrypted at rest
0 Shared spreadsheets in use
Full Audit trail compliance

VaultBridge eliminated all unencrypted credential storage across the organisation. Engineers now access client secrets through a single, audited interface rather than hunting through emails and shared files. Onboarding new engineers takes minutes — grant the appropriate tenant roles and they're ready. Offboarding is equally instant: revoke access, and the audit log confirms no lingering exposure.

Lessons Learned

Building for MSPs means building for trust. The multi-tenant isolation model had to be bulletproof — any possibility of cross-tenant data leakage would undermine the entire value proposition. Enforcing tenant boundaries at the data layer (not just application logic) was a deliberate architectural choice that added complexity but provided genuine security guarantees.

Microsoft SSO integration proved critical for adoption. Engineers are far more likely to use a tool daily when it doesn't require yet another set of credentials. Reducing friction at the authentication layer directly increased engagement across the team.
